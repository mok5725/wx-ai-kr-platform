Karpathy Guidelines

좋은 코딩 습관 4원칙

Andrej Karpathy가 관찰한 AI 코딩의 흔한 함정("모델은 확인 없이 잘못된 가정을 밀어붙이고, 코드를 과하게 복잡하게 만든다")을 막는 4가지 행동 규칙을 클로드에 심습니다. 65줄 남짓의 아주 가벼운 스킬입니다.

결과를 안정시키는 4원칙의 의미·설치·적용 방식.

4원칙 (자세히)

"모델의 가장 흔한 실수는 사용자를 대신해 잘못된 가정을 하고 확인 없이 밀고 나가는 것이다. 또한 코드와 API를 지나치게 복잡하게 만들고, 죽은 코드를 정리하지 않는다." — Andrej Karpathy
1
생각 먼저 (Think Before Coding)

가정을 밝히고, 불확실하면 묻는다. 해석이 여럿이면 조용히 고르지 말고 제시한다. 더 단순한 방법이 있으면 말한다. → 눈치보며 넘겨짚기 대신 사전 확인.

2
단순함 우선 (Simplicity First)

문제를 푸는 최소 코드만. 요청 안 한 기능·추상화·"유연성" 금지. "200줄인데 50줄이면 되면 다시 써라." → 과설계 방지.

3
최소 수정 (Surgical Changes)

고쳐야 할 곳만 건드린다. 옆 코드·서식을 "개선"하지 않는다. 기존 스타일을 따른다. 바뀐 모든 줄은 사용자의 요청으로 설명돼야 한다. → 드라이브바이 리팩터·범위 확장 방지.

4
목표와 검증 (Goal-Driven Execution)

막연한 지시를 검증 가능한 목표로 바꾼다. "버그 고쳐"→"버그를 재현하는 테스트를 쓰고 통과시켜라". 다단계는 단계마다 "→ 검증: [확인]"을 붙인다. → 스스로 반복 검증 가능.

장점 · 단점

👍 장점

아주 가볍고(65줄) 의존성 0·감사 쉬움. 실제로 자주 나는 실수를 정조준. 검증 가능한 목표 프레이밍으로 자가 검증. 지저분한 diff·불필요한 리팩터 감소. CLAUDE.md·플러그인·Cursor로 이식 가능.

👎 단점

"속도보다 신중"에 치우쳐 사소한 일엔 질문이 많아질 수 있음. 개발·테스트 맥락 중심(비코드 작업엔 덜 맞음). 강제가 아닌 지침 → 모델이 덜 지킬 수도. 기존 프로젝트 규칙과 병합·충돌은 사용자 책임.
